# Rituals plugin plan

A companion repository, distributed as a Claude Code plugin marketplace, that layers opinionated engineering rituals on top of `aiwf`. Drafted 2026-04-28 during dogfooding of session-4-complete aiwf; not yet started.

This document captures the architecture we agreed on, the porting roadmap, and the open questions. It supersedes any prior conversation about the rituals layer.

---

## Why this exists

`aiwf` is the planning kernel: six entity kinds, stable ids, validators, git-trailer history, pre-push hook. It deliberately has no opinion about how the work *gets done* — branching, TDD, code review, milestone lifecycle, release ceremonies, tracking docs. Those rituals are valuable, but they're opinions, and forcing them into the kernel would violate aiwf's narrow-scope contract.

The earlier framework, `ai-workflow` (v1, at `~/Projects/ai-workflow/`), bundled all of those rituals into a single submodule-and-`sync.sh` system. That bundling is what motivated `aiwf` in the first place: too much in one box, too coupled, too opinionated by default.

The right shape: keep `aiwf` narrow, ship the rituals separately, let the consumer install both (or just one) and let Claude Code's plugin/marketplace mechanism handle distribution.

## Architecture

### Three repos, three roles

| Repo | Role | Distribution |
|---|---|---|
| `ai-workflow-v2` (this repo) | aiwf core: Go binary + embedded `aiwf-*` skills | `go install` |
| `ai-workflow-rituals` (new) | Marketplace exposing two plugins | `/plugin marketplace add` |
| consumer repo | Both installed | per-consumer setup |

### Three namespaces, three meanings

| Prefix | Owner | Coupling to aiwf | Survives an aiwf swap? |
|---|---|---|---|
| `aiwf-*` | aiwf core (this repo) | IS aiwf | no — replaced with the new kernel's prefix |
| `aiwfx-*` | `aiwf-extensions` plugin | calls `aiwf <verb>`; uses aiwf entity vocabulary | no — every skill rewrites for the new kernel |
| `wf-*` | `wf-rituals` plugin | none — disciplined to stay aiwf-free | yes — the same plugin works against any planning kernel |

The `wf-*`-survives-an-aiwf-swap property is a discipline, not a guarantee. Every `wf-*` skill must be reviewed during the port for `aiwf` shell calls, `D-NNN`/`M-NNN` ID assumptions, aiwf status vocabulary, and aiwf-specific path assumptions. If any are present, the skill belongs in `aiwfx-*`.

### Two plugins in the marketplace

```
ai-workflow-rituals/
├── .claude-plugin/marketplace.json
├── plugins/
│   ├── aiwf-extensions/
│   │   ├── .claude-plugin/plugin.json
│   │   ├── skills/aiwfx-*/SKILL.md
│   │   ├── agents/{planner,builder,reviewer,deployer}.md
│   │   └── templates/{epic-spec,milestone-spec,adr,decision,tracking-doc}.md
│   └── wf-rituals/
│       ├── .claude-plugin/plugin.json
│       ├── skills/wf-*/SKILL.md
│       └── templates/                  # only if wf-quality-score lands here
├── README.md
└── CHANGELOG.md
```

The split lets users opt out: `aiwf-extensions` is the milestone-lifecycle pack; `wf-rituals` is the generic engineering rituals (TDD, code review, doc hygiene) — useful even on a project that doesn't use aiwf.

### Where things land at runtime

| Asset | On consumer machine | Lifecycle |
|---|---|---|
| aiwf binary | `~/go/bin/aiwf` | `go install` updates |
| aiwf core skills (`aiwf-*`) | `.claude/skills/aiwf-*/SKILL.md` in consumer (gitignored) | regenerated by `aiwf init` / `aiwf update` |
| Plugin contents | `~/.claude/plugins/cache/<plugin>/<version>/` | `/plugin update` or auto |
| User-authored skills | `.claude/skills/<your-name>/` in consumer | the user's choice (commit or gitignore) |

The plugin install path is **separate** from `.claude/skills/`, so the user's own skills coexist with no collision risk. This removes v1's `.ai-repo/skills/` workaround entirely — that pattern existed only because v1 wiped `.claude/skills/` on every sync.

## Coupling boundary (the discipline)

A `wf-*` skill must pass this test: *"if I deleted aiwf entirely tomorrow, would this skill still make sense?"* If no, it belongs in `aiwfx-*`.

Concretely, a `wf-*` skill must not:
- Shell out to `aiwf` for any verb.
- Reference aiwf id formats (`E-NN`, `M-NNN`, `ADR-NNNN`, `D-NNN`, `G-NNN`, `C-NNN`).
- Use aiwf status vocabulary (`proposed`, `active`, `done`, `cancelled`, `in_progress`, `accepted`, `rejected`, `superseded`, `addressed`, `wontfix`, `released`, `retired`).
- Assume aiwf's directory layout (`work/epics/...`, `docs/adr/...`, etc.).
- Mention aiwf, the kernel, or any aiwf-specific concept.

A `wf-*` skill *may*:
- Speak the language of the work (milestone, acceptance criteria, branch, test, PR, release).
- Read frontmatter abstractly ("the spec's `id` field") without asserting a format.
- Reference paths abstractly ("the active spec") and let the agent / orchestrator resolve.

The whole `aiwfx-*` namespace is the place for aiwf-specific orchestration. It can call `aiwf promote`, `aiwf add adr`, `aiwf history`, etc. freely.

## Sequencing

### Step 1 — preparatory work in `ai-workflow-v2`

Mechanical changes inside this repo before any rituals work begins. Estimate: 30–60 min.

1.1. Rename `wf-*` → `aiwf-*`:
- `tools/internal/skills/wf-*/SKILL.md` directories → `aiwf-*/`.
- Embed paths and skill-name strings in the binary (skills, initrepo, doctor packages).
- `.gitignore` append paths in `tools/internal/initrepo/`.
- `aiwf doctor` byte-comparison logic that points at the embedded copies.
- Tests that assert on `wf-*` paths or directory names.
- README `§"Coexistence with your .claude/"` rewording.
- Pre-push hook marker `# aiwf:pre-push` stays unchanged (orthogonal).

1.2. Drop `wf-track` from `docs/poc-plan.md` session 5. The tracking-doc convention moves to `aiwfx-track` in the rituals plugin.

1.3. Update `README.md` `§"Coexistence"`:
- aiwf core only touches `aiwf-*` (the new prefix).
- `aiwfx-*` and `wf-*` are namespaces used by the companion rituals plugin, not by aiwf core.
- The rituals plugin is an optional separate install (point at `ai-workflow-rituals` once it exists).

1.4. Verify `aiwf doctor --self-check` still green; commit.

### Step 2 — scaffold `ai-workflow-rituals`

New repo. Marketplace shape established before any skills land.

2.1. Repo skeleton:
- `.claude-plugin/marketplace.json` listing both plugins (with `source` paths under `plugins/`).
- `plugins/aiwf-extensions/.claude-plugin/plugin.json` (minimal — `name`, `description`).
- `plugins/wf-rituals/.claude-plugin/plugin.json` (minimal).
- Empty `skills/`, `agents/`, `templates/` directories under each plugin.
- `README.md` with install commands and the coupling-boundary statement.

2.2. Validate end-to-end before any skill content:
- `/plugin marketplace add <repo>` from a sandbox.
- `/plugin install` for both plugins.
- Confirm Claude Code picks up the (empty) plugin manifests.

This is the **first validation point**. Architecture is real before any porting work.

### Step 3 — port `wf-rituals` (lower friction; no aiwf coupling)

Skills to port from v1 (`~/Projects/ai-workflow/skills/`):

| Skill | v1 source | Port effort | Notes |
|---|---|---|---|
| `wf-patch` | `patch.md` | trivial | Pure git ritual; ~zero changes. |
| `wf-tdd-cycle` | `tdd-cycle.md` | small | Drop v1-specific tool refs; keep the red-green-refactor + branch-coverage discipline. |
| `wf-review-code` | `review-code.md` | small | Drop v1 status vocabulary; keep the checklist. |
| `wf-doc-lint` | `doc-lint.md` | medium | Port mechanical-checks subset only; skip v1's contract-surface logic until aiwf has a story for it. |

After 3.x: install into a test consumer, run `wf-patch` and `wf-tdd-cycle` end-to-end. **Second validation point.**

### Step 4 — port `aiwf-extensions`

Templates first, then skills, then agents.

4.1. Templates:
- `templates/epic-spec.md` (rich)
- `templates/milestone-spec.md` (rich)
- `templates/tracking-doc.md`
- `templates/adr.md` (Nygard-shaped, prose-heavy)
- `templates/decision.md` (new — minimal, matches aiwf's `D-NNN` body skeleton)

aiwf core's `BodyTemplate` (in `tools/internal/entity/serialize.go`) stays unchanged; it provides the empty shells. These richer templates fill them in.

4.2. Skills:

| Skill | v1 source | Port effort | Notes |
|---|---|---|---|
| `aiwfx-plan-epic` | `plan-epic.md` | medium | Status vocab + replace `wf-graph promote` with `aiwf promote`. Use aiwf's `E-NN` allocation. |
| `aiwfx-plan-milestones` | `plan-milestones.md` | medium | Same pattern. |
| `aiwfx-start-milestone` | `start-milestone.md` | medium | Replace `wf-graph promote` with `aiwf promote`; use `aiwfx-track` to scaffold tracking doc. |
| `aiwfx-wrap-milestone` | `wrap-milestone.md` | medium-large | Status vocab + contract verification driving aiwf's `contract` entity kind. |
| `aiwfx-wrap-epic` | `wrap-epic.md` | medium | Status vocab + ADR harvesting via `aiwfx-record-decision`. |
| `aiwfx-release` | `release.md` | small | Status vocab; semver/tag ritual unchanged. |
| `aiwfx-track` | new | small | Describes the tracking-doc convention. Path: `work/tracking/<M-id>-<slug>.md`. Advisory only. |
| `aiwfx-record-decision` | replaces v1's `architect.md` | small | Thin recipe (~25 lines): pick ADR vs. D-NNN, allocate, fill template, commit. Invoked in-flow from any skill. |

4.3. Agents (4 thin role personas, ~50 lines each):

| Agent | Skills it drives |
|---|---|
| `planner` | `aiwfx-plan-epic`, `aiwfx-plan-milestones`, `aiwfx-record-decision` |
| `builder` | `aiwfx-start-milestone` → `wf-tdd-cycle` → `aiwfx-wrap-milestone` |
| `reviewer` | `wf-review-code`, status promotions via `aiwf promote` |
| `deployer` | `aiwfx-release` |

Agents are bundled with `aiwf-extensions` because they reference aiwf entity vocabulary. They're written thinly: the heavy ritual logic lives in the skills, not the agent prose.

After 4.x: install into a test consumer, drive an end-to-end milestone cycle (`add epic` → `start milestone` → TDD → `wrap milestone`). **Third validation point.**

### Step 5 — `ai-workflow-rituals` README

User-facing docs in the new repo. After both plugins are working:

- What this is, what it complements (`aiwf`).
- Marketplace install command.
- Per-plugin opt-in.
- Coupling boundary statement (so contributors know `wf-*` is aiwf-agnostic by design).
- Pointers to `aiwf` for the planning side.

## What is *not* in scope

Skills from v1 deliberately not being ported, with reasoning:

| v1 skill | Why not |
|---|---|
| `architect` | Replaced by thin `aiwfx-record-decision`. Heavy authoring rarely happens in practice; decisions surface in-flow. |
| `workflow-graph` | aiwf already does this (validators + history). Porting would duplicate. |
| `update-framework` | v1's submodule-sync mechanism. Plugins handle their own updates via `/plugin update`. |
| `verify-contracts` | Defer. aiwf's `contract` entity kind has artifact validation; richer schema-evolution loops can land later if needed. |
| `design-contract` | Defer. Same reason — bundle as a single ADR + schema + fixtures workflow if real friction shows up. |
| `quality-score` | Defer. Useful but not load-bearing. |
| `dead-code-audit` | Defer. Useful but not load-bearing. |
| `devcontainer` | Out of scope; consumer-specific. |
| `doc-garden` | Defer. Interactive triaging is hard to skill-ify; let the user-AI loop handle it for now. |

If real friction shows up later, port from v1 then. YAGNI.

## Naming and identity

- aiwf's prefix stays `aiwf-*`. Project-name-as-prefix is conventional (`git-*`, `kubectl-*`); the cost of "rename project means rename skills" is one mechanical sweep, paid only if the project is actually renamed.
- `aiwfx-*` reads as "aiwf eXtension." Single-letter difference from `aiwf-*` is fine because skills are AI-invoked, not human-typed.
- `wf-*` is generic and survives kernel swaps; that survival depends on the discipline, not the name.

## Open questions

1. **Repo URL.** `github.com/23min/ai-workflow-rituals`? Or a different name (`aiwf-rituals`, `claude-aiwf-rituals`, `wf-pack`)? Preference noted, not committed.

2. **Auto-update vs. pinned versions.** Per the plugin research, omitting `version` in `plugin.json` makes every commit a new version (auto-update on startup). Worth doing while the rituals repo is evolving fast; pin once stable. Defer the decision until the plugin actually lands.

3. **Should aiwf core's skills *also* ship as a marketplace plugin?** Currently no — the binary's `embed.FS` is the single source of truth. Revisit only if real friction shows up around discovery (e.g., users say "I had no idea aiwf had skills until I ran `aiwf init`").

4. **Tracking docs convention specifics.** `aiwfx-track` will define the path (`work/tracking/<M-id>-<slug>.md`) and section structure (acceptance criteria checklist, learnings, blockers, decisions). The exact section list is a TBD during the port — base it on v1's `templates/tracking-doc.md` but trim aggressively.

5. **agent-history files.** v1 keeps `work/agent-history/<agent>.md` for accumulated learnings. User confirmed: keep. Where does the convention live — in the agent prose ("append your learnings here at end of work"), or as a small `aiwfx-history` skill? Lean: agent prose. Skill is overkill for "append a paragraph."
