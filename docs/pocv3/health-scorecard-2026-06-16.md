## Summary

The aiwf kernel is in strong overall health: 12 of 24 principles verdict Strong, 11 Weak, 1 Missing, with the codebase's defining strength being its mechanical-enforcement story — 47 CI-blocking Go tests under internal/policies/ plus pre-commit/pre-push aiwf check hooks pin most invariants (atomic writes via PolicyAtomicWriteChokepoint, FSM closed-sets via fsm_invariants.go, trailer/audit shape, no-silent-fallback, no-retry-on-git, finding-hint completeness, skill coverage) so that correctness does not depend on reviewer or LLM recall. The Weak verdicts concentrate in two clusters. First, an enforcement-gap cluster where the property holds in the current tree by inspection but has no chokepoint: A3 layering (no depguard/import-direction test — an acyclic upward edge would compile and ship), C1 single-source-of-truth (the aiwf version verb and JSON envelope can report different strings for an ldflags-stamped binary), and parts of B3/F1/F2 where documented conventions (no_silent_fallback's bare-identifier blind spot, enums:ignore rationale-required, no-TODO/nolint bans) are advisory rather than mechanical. Second, a test-architecture cluster: D2 seam-equivalence (PageDataResolver and BranchOracle have divergent implementations with no shared conformance suite — G-0222 open), D3 branch-coverage (the lone Missing — CI prints a non-blocking coverage summary with no diff-scoped gate, G-0067 open), and D4 altitude (no cross-impl resolver equivalence at unit level; htmlrender substring assertions that the repo's own convention forbids). E1 structured-logs is Weak because the JSON envelope is genuinely Strong structured output but the diagnostic surface is 239 bare fmt.Fprint(os.Stderr) sites with zero slog, directly contradicting CLAUDE.md:478 (ADR-0017 proposed, G-0223 open). B2 (non-strict manifest/config), C4 (entity frontmatter carries no version field), C2 (two coexisting convergence idioms, no cross-verb idempotence policy), F3 (decision-to-decision supersession is prose-only and unmechanized, ADR-0016 has live frontmatter/body status drift), and G2's advisory what-undoes-this gate round out the Weak set.

### Delta from the 2026-06-04 audit

Net movement is positive: C3 Atomic writes flipped Weak → Strong, reflecting landed work — G-0221's central pathutil.AtomicWriteFile helper (commit 1aba9334) plus PolicyAtomicWriteChokepoint now bans raw os.WriteFile/os.Create in production code and is wired into the CI test run, resolving clean against the live tree. The offsetting regressions are three principles that dropped Strong → Weak: A3 Layered, C1 Single source of truth, C2 Idempotence, and C4 Versioned schemas — these were not new failures but tighter adversarial scrutiny finding that each holds only by inspection or per-verb test with no dedicated chokepoint (A3 has no import-direction policy; C1 surfaced the version-source split between aiwf version and the JSON envelope; C2 has no cross-verb idempotence assertion; C4 noted entity frontmatter lacks a version field). The three Weak verdicts carried over unchanged: D2 Equivalence-at-seams (G-0222 still open, no progress this window), D3 Branch-coverage which remains the lone Missing (G-0067 still open, CI coverage still non-blocking), and E1 Structured-logs (ADR-0017 still proposed, G-0223 still open, zero slog in production). Net Strong count moved from 20 to 12, but the drop is dominated by stricter rubric application to enforcement-gaps rather than code regressions — the only genuine code-quality change is C3's improvement.

### A2. Low coupling

- **Verdict:** Strong
- **Summary:** The internal dependency graph is a clean DAG with zero cycles (verified by DFS over `go list -json ./internal/...`; Go also forbids cycles at compile time). Layering is strict and unidirectional: `entity → codes`; `tree → entity, trunk`; the validator core `check → entity, gitops, scope, tree, codes`; `verb` fans into the lower layers; and every `cli/<verb>` subpackage depends on `cli/cliutil` plus lower packages but never the reverse. `cliutil` is imported by 29 packages yet all are CLI-layer peers/siblings — no lower layer (entity, tree, check, verb, gitops, scope, config, pathutil, codes, render, htmlrender, trunk) imports it, so the principle's specific prohibition (no junk-drawer hub imported by lower layers) holds. The two cross-impl seams are narrow, named, well-documented interfaces that invert dependencies cleanly: `check.BranchOracle` (the pure check rule defines it; the CLI gather layer at `internal/cli/check/` supplies the git-backed impl, keeping the rule free of live-git coupling) and `htmlrender.PageDataResolver` (htmlrender stays decoupled from the live tree). Mutable global state is minimal and each instance is justified — an `-ldflags` version stamp (`internal/cli/root.go:64`), a documented cli↔doctor cycle-breaker seam (`internal/cli/doctor/dispatcher.go:16`, wired once in `root.go:72`), and an unmockable-syscall wrapper (`internal/cli/upgrade/upgrade.go:425`); all remaining package vars are read-only lookup tables (FSM transition maps, id patterns, schemas) or embedded FS. The one genuine gap: unlike nearly every other invariant in this repo, the coupling/layering direction is NOT mechanically pinned — there is no depguard/gomodguard linter and no architecture-layering policy test among the 47 in internal/policies/.
- **Evidence:**
  - supports: `internal/check/isolation_escape.go:151` BranchOracle is a narrow named interface (FirstParentBranches/OracleErrors/BranchOfSHA) defined in the pure check package; the git-backed impl gitBranchOracle lives in internal/cli/check/isolation_escape_oracle.go.
