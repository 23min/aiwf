# M-075 — Lifecycle-gate entity-body-empty rule (closes G-071)

## Goal

Make `internal/check/entity_body.go`'s `entity-body-empty` rule lifecycle-aware: skip terminal-status entities (any kind) and skip ACs whose parent milestone is `draft`. Closes G-071's two cases — pre-implementation noise from freshly-allocated ACs in draft milestones, and post-implementation noise from preserved terminal-status entities like ADR-0002.

## Context

The rule was scoped at M-066/AC-1 to catch *shipped* empty bodies — entities promoted to terminal status without prose — but the implementation generalises that intent to every lifecycle state. Two false-positive bands surround the active-state target population: pre-implementation drafts (24 warnings on E-20's M-072/M-073/M-074 today, generated by `aiwfx-plan-milestones`) and post-implementation terminal artifacts (3 standing warnings on ADR-0002 alone). G-071 documents both cases; this milestone fixes both with a single status-gating predicate. The `entity.IsTerminal(kind, status)` helper is the load-bearing primitive — named verbatim in proposed ADR-0004 §Trigger; coordinated with E-20/M-072 (whichever epic ships first adds the helper, the other consumes).

## Constraints

- Status gating only. Phase gating (skipping ACs based on `tdd_phase: red` plus parent milestone status) is out of scope per the epic spec — defer until precision-need justifies the extra complexity.
- The helper `entity.IsTerminal(kind, status)` is a pure closed-set switch on `entity.Kind`. Returns `true` for the kind's terminal status set: `{cancelled, done}` for epic/milestone, `{superseded, rejected}` for ADR/decision, `{addressed, wontfix}` for gap, `{retired, rejected}` for contract.
- Coordinate with E-20/M-072 on the helper — it lives in `internal/entity/transition.go`. If M-072 has already added it, M-075 consumes; if M-075 ships first, M-075 adds and M-072 consumes.
- The active-state population (entities in non-terminal, non-draft states with empty load-bearing sections) MUST still trigger the rule. AC-4 is the regression guard.
- Net change in `internal/check/entity_body.go` is small (~10–20 lines): two predicate calls in the existing rule body. No restructuring of the per-kind dispatch.
- TDD-required: each AC drives a red→green→refactor cycle.

## Design notes

- The rule walks every entity once (already does). Two added checks:
  1. For top-level body sections (per `requiredSectionsByKind`): skip the entity entirely if `entity.IsTerminal(e.Kind, e.Status)` returns true.
  2. For AC sub-element bodies (under a milestone parent): skip if the *parent milestone's* status is `draft`. The parent milestone is the entity being walked at the call site already — read `e.Status == entity.StatusDraft` as the gate.
- The two gates are independent. A terminal milestone with `draft` parent? Impossible (parent is `e` itself; `e` is the milestone). The conditions don't overlap.
- ADR-0002's `## Context`, `## Decision`, `## Consequences` are top-level sections; the terminal-status gate (AC-2) is what silences those. M-072/M-073/M-074's empty AC bodies are AC sub-element sections; the draft-parent gate (AC-3) is what silences those.
- AC-5's "warning baseline drops by 27" is the integration check: 24 from E-20's milestones + 3 from ADR-0002. Counted on the `poc/aiwf-v3` tree as it stands at M-075's start.

## Surfaces touched

- New tests in `internal/check/entity_body_test.go` — fixtures for the two exemption paths and the active-state non-exemption.

## Out of scope

- Phase gating (`tdd_phase`-aware skipping for ACs in `tdd: required` milestones).
- Any change to which sections are required per kind (`requiredSectionsByKind` table).
- Per-kind body-section validation expansion (e.g., adding `finding` once F-NNN ships — that's E-19/follow-up).
- Coordination with ADR-0004's archive-convention (when ADR-0004 lands, the rule could also skip files under `archive/` paths; that's a separate, additive change).
