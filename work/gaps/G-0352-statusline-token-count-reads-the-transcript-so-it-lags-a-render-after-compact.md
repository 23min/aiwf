## Problem

The aiwf statusline's token segment (`▸ <tokens>`) does not drop immediately after
`/compact`. The operator compacts to shrink the context, but the statusline keeps showing
the pre-compaction token count until the *next* assistant turn, then corrects. It reads as
"the statusline is stale / didn't refresh."

It is not a Claude Code refresh problem. Claude Code re-runs the statusline command after
`/compact` finishes (it is a documented trigger event, alongside per-assistant-message and
the optional `refreshInterval` timer). The staleness is in how the aiwf statusline *derives*
the number.

## Root cause

The `--- Token usage from transcript ---` block in the embedded statusline walks the
transcript file bottom-up and takes the **most recent assistant `usage` block**:

```
tokens="$({ tail -r "$transcript" ... } | jq -r 'select(.message.usage != null)
         | .message.usage | (.input_tokens // 0) + (.cache_read_input_tokens // 0)
         + (.cache_creation_input_tokens // 0)' | head -n 1)"
```

`/compact` does not append a fresh assistant `usage` entry for the compacted context, so the
most-recent `usage` block is still the *pre-compaction* turn — its `input_tokens +
cache_read_input_tokens + cache_creation_input_tokens` equal the full old context size. The
statusline re-runs at `/compact` but reads the same stale block, so nothing visibly changes.
It self-corrects on the next assistant turn, which is sent with the compacted context and
writes a small `usage` block. Net effect: stale for exactly one render, not permanently
stuck.

The stdin JSON that Claude Code passes the statusline carries a `context_window` object
(`total_input_tokens`, `context_window_size`, `used_percentage`, `current_usage{…}`), but the
script uses it only for the 7d/5h rate-limit dots — never for the token/context ball, which
is recomputed from the transcript instead.

## Proposed direction — source tokens from the stdin `context_window` field

Switch the token count (and the percent that colors the ball) from the transcript walk to
the stdin `context_window` field: `context_window.total_input_tokens` for the count,
`context_window.used_percentage` (or `total_input_tokens / context_window_size`) for the
color threshold. This field reflects the *current* context window and landed in Claude Code
v2.1.132 (before that it was a cumulative session total); the repo's own environment is well
past that (observed v2.1.199).

Bonus simplification if the swap holds: it removes the transcript read, the `tail -r` / `tac`
BSD-vs-GNU portability dance, and the manual `ctx_max` 1M/200k detection (use
`context_window_size` directly).

## The immediacy caveat — no longer a precondition for the swap

The original framing gated the swap on proving it fixes the `/compact` staleness: Claude
Code docs state `context_window.current_usage` is `null` after `/compact` until the next API
call repopulates it, and the compaction summarization call itself reads the *full*
pre-compaction context — so the stdin `context_window` field may lag exactly like the
transcript walk does, in which case the swap would be a cleanup, not a fix for the reported
symptom.

Re-evaluated: that risk doesn't justify blocking the swap. `context_window` is Claude Code's
own accounting of the *current* session (this repo is well past v2.1.132), so it is at worst
equivalent to the transcript walk and at best fixes the staleness — either way strictly
better than today, while also removing the transcript read, the `tail -r`/`tac`
BSD-vs-GNU dance, and the manual `ctx_max` 1M/200k string-sniffing. **Make the swap
unconditionally**; treat the live-`/compact` immediacy observation as evidence to capture
opportunistically during or after implementation, not a gate before starting.

## Build requirements

- **Author location:** `internal/skills/embedded-statusline/statusline.sh` (the embedded
  source; `internal/skills/statusline.go` embeds it; `aiwf init` / `aiwf update` materializes
  it to `~/.claude/statusline.sh`).
- **Do not hand-edit the materialized `~/.claude/statusline.sh`** — its header declares it
  aiwf-managed and regenerated by `aiwf update`, which overwrites local edits. The fix lives
  in the embedded source only.
- **Behavioral tests** already exist at `internal/policies/statusline_behavioral_test.go`
  (they feed synthetic stdin JSON and assert the rendered line). Extend them: a
  `context_window`-bearing stdin renders the expected token count/color, and the color
  thresholds track `used_percentage` / `context_window_size`. Follow the existing
  synthetic-JSON harness in that file rather than reading a real transcript.
- Keep the fail-soft contract: a missing/malformed `context_window` field must degrade to
  a sensible default (the current transcript path is one candidate fallback), never a hard
  error in the render.

## Suggested acceptance criteria for the implementing change

1. With a `context_window` object on stdin, the token segment renders `total_input_tokens`
   (formatted) and the ball color tracks `used_percentage` / `context_window_size`; asserted
   in `statusline_behavioral_test.go`.
2. Absent or malformed `context_window` degrades to the fallback path without erroring;
   asserted by a defensive-input test.
3. (Optional, non-blocking) If a live `/compact` is observed on the target version during
   implementation, record whether the token segment updates at the compact render or one
   turn later — nice-to-have evidence, not a precondition for landing the change.
4. The full statusline behavioral suite and `aiwf check` are clean.
