## What's missing

`cmd/stresstest`'s `verb-sequence` scenario drives a random walk of `aiwf promote` calls against every entity kind and judges each outcome against `entity.ValidateTransition`'s bare FSM legality, plus a standing assertion that `aiwf check` never regresses beyond a curated baseline of expected warnings (`classifyVerbSequenceStep` and `verbSequenceExpectedWarnings`, `internal/stresstest/verb_sequence.go`). That second assertion is generic — any `aiwf check` finding outside the baseline is a violation — so it transitively picks up any *standing check rule* that later gets added for a bad state a verb-time guard alone doesn't cover. It does not, on its own, assert that a domain-specific verb-time guard (as opposed to a background check rule) exists at all: a verb-time guard missing entirely, with no check-rule backstop either, is invisible to it — a silent bypass looks identical to a legitimate success.

Confirmed empirically against G-0335 (`aiwf promote <milestone> cancelled` used to bypass the open-AC guard `aiwf cancel` already enforced): running `verb-sequence` 30x against the pre-fix binary passed 30/30 with zero violations. G-0335's fix closed two things: the missing verb-time guard itself, and a companion standing check rule (`milestone-cancelled-incomplete-acs`) that backstops any future regression of that guard — which `verb-sequence`'s existing generic check-clean assertion now catches automatically, no scenario code change needed.

That resolves the immediate instance for `verb-sequence`, but the underlying architectural gap is broader on two axes:

1. **Other scenarios don't share `verb-sequence`'s generic oracle.** The concurrency-focused scenarios that call `aiwf check` at all (`parallel-branch-reallocate`, `cross-worktree-id-race`, `force-override-durability`, `promote-on-wrong-branch-detection`, `reachability-isolation`) each assert on exactly one scenario-specific finding code, not "no unexpected finding" — so a check-rule regression these scenarios happen to produce as a side effect would go unnoticed even though `aiwf check` was run.
2. **No scenario exercises concurrent promote/cancel/AC races at all.** `verb-sequence` walks one process against its own disposable repo, sequentially. None of the true concurrency scenarios manipulate ACs, promote, or cancel — so a race between two worktrees (e.g. one promoting an AC while another cancels its parent milestone) is untested territory regardless of oracle breadth.

## Why it matters

The stress catalog is the repo's only automated surface driving `aiwf` end-to-end under randomized/concurrent state; a reader could reasonably assume a clean catalog run means state stays coherent under any plausible sequence or interleaving of verb actions. It doesn't fully test that yet. A guard added to one verb surface but not its sibling — G-0335's shape — now gets caught for `verb-sequence` for free going forward (via the check-rule backstop pattern), but the same class of bug reached purely through *concurrent* interleaving has no equivalent safety net, and a check-rule regression specific to one of the narrow-oracle scenarios would still ship silently.

Two concrete, complementary fix directions, not mutually exclusive:

- **Broaden the "check must stay clean" oracle onto the scenarios whose end-state is expected to be a coherent, loadable tree** — `parallel-branch-reallocate`, `cross-worktree-id-race`, `force-override-durability`, `promote-on-wrong-branch-detection`, `reachability-isolation`, `archive-during-active-scope`, `cross-worktree-edit-body-race`, `concurrent-move`, `concurrent-writer-at-scale`, `concurrent-id-allocation` — each with its own curated expected-warnings baseline mirroring `verbSequenceExpectedWarnings`, not a shared one (different scenarios produce different incidental noise). Deliberately excludes `disk-fault`/`lock-kill`/`mid-write-kill`/`head-drift`, whose whole point is a torn-write or crash-recovery intermediate state `aiwf check`'s vocabulary doesn't model.
- **Give `verb-sequence` a concurrent mode**: race N walkers against the same shared entities instead of each in its own isolated disposable dir, reusing `concurrent-writer-at-scale`'s existing subprocess-fan-out harness (`internal/stresstest/concurrent_writer_at_scale.go`'s goroutine + `sync.WaitGroup` pattern launching real `aiwf` subprocesses against one shared repo dir) — this is the harness shape needed, not a bespoke new scenario. Reuses `verb-sequence`'s existing walker and oracle almost unchanged; the delta is pointing N of them at shared state instead of isolated state.
