## What's missing

There is no surface that joins git's worktree state with aiwf's entity state. When the operator runs multiple sessions across multiple worktrees â€” one driving E-NNNN, another driving G-NNNN via `wf-patch`, a third closing a milestone â€” the planning tree knows which entities are `in_progress` and git knows which worktrees and branches exist, but nothing renders the join.

`aiwf status` enumerates in-flight epics, milestones, and gaps, but is blind to which worktree (if any) is currently driving each entity, what's already merged to trunk versus still-ahead on a feature branch, and what's queued behind each in-flight item via `depends_on` edges. The "if this finishes, what unblocks next?" view does not exist on any surface today.

## Why it matters

Parallel workstreams are the explicit working mode of this project â€” trunk-based development with per-topic worktrees, periodically wrapping and interleaving work between separate epics and gaps. The operator pays a coordination tax on every context switch: re-deriving "what's where" from `git worktree list`, branch-name guesses, scanning `aiwf status`, and chasing `depends_on` edges by hand. The cost compounds when wraps interleave between concurrent epics.

The framework's stated goal is to keep humans and AI assistants tracking what's planned, decided, and done. The *in-flight slice* of that picture â€” with worktree provenance and downstream visibility â€” currently lives only in the operator's head. The data to project it is already present (git worktree list, frontmatter status, branch divergence vs. trunk, `depends_on` edges); the gap is that no verb joins them.

## Considered and rejected during discussion

A tracked `in-flight.md` (or `progress.md`) regenerated by a post-commit hook. Rejected for three reasons:

1. **Merge-conflict generator.** A file rewritten by every commit, in a workflow whose explicit purpose is parallel branches, will conflict on every merge â€” the artifact intended to reduce coordination overhead becomes its own coordination tax.
2. **Conflicts with the marker-managed pattern.** Per CLAUDE.md, framework artifacts in the consumer repo are gitignored skills + untracked hooks, regenerated only on explicit `aiwf init` / `aiwf update`. A hook-written tracked file is a third pattern with no existing precedent.
3. **"What verb undoes this?" has no clean answer.** A commit-driven side-effect file has no inverse; reversal would mean hand-editing or re-running.

## Likely resolution shape (forks still open)

A read-only verb that joins:

- `git worktree list` â†’ branch, path, last commit per worktree
- `aiwf list --status in_progress` â†’ entities currently in flight
- worktreeâ†”entity correlation â†’ which worktree drives which entity
- `git log <trunk>..<branch>` â†’ what hasn't merged yet per worktree
- `depends_on` edges â†’ what's queued behind each in-flight entity

Output human-readable to stdout by default; `--format=json` for tooling, per the JSON-envelope convention. No persisted artifact; regenerated on demand like `aiwf status`. A `--watch` mode and an HTML section in `aiwf render` are natural follow-ons once the data shape settles.

Design forks still open:

1. **Verb vs. extension of `status`.** A new `aiwf inflight` is more discoverable; extending `aiwf status` with `--worktrees` keeps the surface small. ADR-0006's skills policy nudges toward extending an existing verb unless prompt shapes diverge enough.
2. **Worktreeâ†’entity inference.** Branch-name parsing is fragile. A commit trailer or a small `.aiwf/worktree-scope` marker written by `aiwfx-start-milestone` (and equivalent ritual entry points) would be more honest. The marker option requires ritual-side cooperation; the trailer option already has a precedent in the `aiwf-entity:` trailer chain and could be inferred from recent commits on the worktree's branch.
3. **CLI-only first vs. simultaneous HTML view in `aiwf render`.** CLI first lands the join logic cheaply; the HTML view is the natural second step once the data shape is pinned. Doing both at once risks committing to the rendered shape before the verb's output is settled.

The "downstream `depends_on`" piece is the most novel â€” it's not surfaced anywhere today and is the real cognitive-load reducer for sequencing decisions, independent of which rendering choice wins.
