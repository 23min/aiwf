## What's missing

aiwf materializes content into the consumer repo via two surfaces only:

- `.claude/skills/aiwf-*` (the verb skills, the `aiwfx-*` ritual
  skills, the `wf-*` engineering skills, the role agents, the
  entity templates) — written and refreshed by `aiwf init` and
  `aiwf update`, gitignored, byte-refreshed on every update.
- `.git/hooks/<hook>` (pre-commit, pre-push, commit-msg) —
  written by `aiwf init` and `aiwf update`, identified by an
  `# aiwf:<hook>` marker so user-written hooks are left alone.

Both surfaces are off-trunk and aiwf-owned by convention. The
consumer's `CLAUDE.md` at the repo root is *user-owned* — aiwf
has no write channel into it today.

The consequence: kernel-level advisory guidance that aiwf wants
to communicate to LLM agents working in consumer trees has no
auto-refreshing home. It ships in skill bodies (loaded only when
the skill is invoked), but the most load-bearing rules — the
ones that need to survive `/compact`, that need to anchor every
turn regardless of what skill is in play — have no surface that
re-anchors per turn.

The G-0242 fix lifted the gate-discipline rule into this repo's
own `CLAUDE.md` (re-injected per turn, survives `/compact`).
That fix works *here*. Downstream consumers running `aiwf init`
or `aiwf update` get the skill bodies refreshed — but the
standing-invariant framing in the skill preambles loads only
when the skill is. They do not get a `CLAUDE.md`-level rule.

## Why it matters

The kernel principle "framework correctness must not depend on
LLM behavior" applies inversely to advisory guidance: the
guidance is the only correctness mechanism we have for rules
that cannot be mechanically enforced (per-action gate
discipline, conversational discipline, the "never suggest the
user pause" rule, the gate against fabricated id-shapes). For
those rules to bind reliably across a long-running session
through multiple `/compact` cycles, they must live in a surface
the LLM sees on every turn.

Today the only such surface in a consumer tree is the
consumer's own `CLAUDE.md` — which aiwf does not write to. So
the strongest LLM-binding channel is the one channel aiwf
cannot reach.

The blast-radius asymmetry is real: a consumer that adopts aiwf
gets the mechanical layer (verbs, checks, hooks, FSM
enforcement) plus the easy-to-load skill layer, but loses the
hardest-to-replace layer (the per-turn re-anchored rules).
Without this work, consumer LLM agents drift across compaction
in ways that no aiwf check rule will catch.

## Direction

Two shapes are viable; the recommendation is the first.

**Shape A — marker-managed CLAUDE.md fragment, gated by
ADR-0015-style consent.** `aiwf init` and `aiwf update` offer
to insert (or refresh) a marker-delimited block in the
consumer's `CLAUDE.md`:

```
<!-- aiwf:guidance:START - DO NOT EDIT, regenerated by aiwf update -->
... aiwf-owned guidance ...
<!-- aiwf:guidance:END -->
```

User content outside the markers is preserved verbatim. Edits
between the markers on subsequent `aiwf update` runs are
overwritten — but the marker block itself is *opt-in*: the
first time `aiwf init` / `aiwf update` would touch CLAUDE.md,
the operator gets an interactive `[y/N]` confirm (TTY) or the
explicit `--wire-claudemd` flag is required.

This matches the ADR-0015 pattern already in place for
`settings.json` edits: aiwf does not silently mutate
user-owned files; consent is per-invocation. The consent flow
either composes with the existing `--wire-settings` machinery
or extends it.

The fragment content is kernel-curated and version-pinned to
the embedded-rituals snapshot — the same source of truth that
ships skills. It starts small (the gate-discipline rule from
G-0242, the id-shape rule from G-0184, the conversational
discipline rules); future kernel guidance is added the same
way.

**Shape B — separate `AIWF.md` file at consumer root,
referenced from CLAUDE.md.** aiwf writes its own file,
doesn't touch CLAUDE.md. Less invasive. But it only helps if
the consumer's CLAUDE.md actually references AIWF.md — which
puts us back to "depends on the LLM remembering to read it."
The reference itself can be lost across `/compact` summary
rewrites. Weaker on the dimension that matters.

Shape A is recommended for the same reason hooks live where
they live: the strongest binding is the closest binding.
CLAUDE.md is the load-bearing channel; aiwf should reach it
the same way it reaches `.git/hooks/`.

## Open questions

1. **ADR-0015 reuse vs new ADR.** ADR-0015 ratifies
   per-invocation consent for `settings.json` edits. Extending
   the same principle to `CLAUDE.md` edits is a parallel case;
   the question is whether one ADR-0015 amendment covers both
   or whether `CLAUDE.md` edits warrant their own ADR (the
   risk profiles differ — settings.json is structured and
   tooling-readable, CLAUDE.md is prose).

2. **Default opt-in vs opt-out.** First-run `aiwf init` could
   default to wiring the fragment (with `--no-wire-claudemd`
   to decline) or default to skipping (with `--wire-claudemd`
   to enable). The setting-edits convention is opt-in. The
   marker-managed nature here (user content outside the block
   is untouched) makes opt-in-by-default less risky than for
   settings.json, but the convention should be consistent.

3. **Marker drift policing.** If a consumer deletes the END
   marker by hand, the next `aiwf update` would either
   refuse, recreate, or replace. Each has a failure mode; the
   pre-existing pattern for hook markers handles this — port
   that policy.

4. **Fragment content scoping.** Not every aiwf-shipped skill
   or principle belongs in the fragment. The selection criteria
   need to be explicit: "rules that must survive `/compact` and
   apply across all turns" — not "everything aiwf wants to
   document."

## Test surface

- Consent flow: interactive prompt on TTY, refusal-without-flag
  on non-TTY (matching ADR-0015's settings.json pattern).
- Marker integrity: insert / refresh / preserve-outside-markers
  / handle-missing-end-marker — each as a fixture-tree test.
- Idempotent refresh: re-running `aiwf update` against an
  already-wired CLAUDE.md produces no diff.
- Content-version pin: the fragment carries an aiwf-version
  comment; the embedded-rituals snapshot path constants point
  at the canonical content.
- Cross-package coupling: the wire-CLAUDE.md flag composes with
  the existing wire-settings flag without flag-name collisions.
