## What's missing

`wf-doc-lint` (`internal/skills/embedded-rituals/plugins/wf-rituals/skills/wf-doc-lint/SKILL.md`) runs four mechanical, report-only heuristics — code-reference drift, removed-feature docs, orphan documents, documentation TODOs — over a docs root that defaults to `docs/`. Three mechanical checks that fall squarely inside its charter (deterministic, structural, report-only, no prose rewrite) are absent, and its default scope omits the hand-authored narrative files at the repo root.

1. **Markdown link integrity (the headline gap).** Check #1 verifies that referenced *code symbols* exist, but nothing verifies that *markdown links resolve*:
   - broken intra-repo links — `[x](path/to/doc.md)` where the target was moved or renamed;
   - broken anchors — `[x](other.md#heading)` (or a bare `#anchor`) with no heading slugging to it;
   - broken source-file links — `[loader](../internal/entity/load.go)` where the path no longer exists.

   These are more deterministic than the symbol heuristics already shipped, yet a docs tree dense with relative cross-links passes today with dead links intact.

2. **CLI-invocation resolution in docs.** The repo already enforces that every backticked `` `aiwf <verb>` `` in a *skill* body resolves (`internal/policies/skill_coverage.go`), but `docs/` is not covered by it. A doc showing `aiwf <verb> --old-flag` for a removed verb or flag passes silently. Generalizes to any CLI-tool repo as "invocations of the project's own binary resolve against `--help`."

3. **Structural checks.** Hand-maintained table-of-contents drift (a `## Contents` list that no longer matches the headings below it) and heading-hierarchy sanity (skipped levels, anchor-colliding duplicate headings, empty sections).

4. **Scope widening to root narrative files.** The default docs root stops at `docs/`, but the front-door narrative — `README.md`, `CONTRIBUTING.md` — lives at the repo root and is the densest code-referencing surface. Widen the default to "the narrative docs root (`docs/`) plus hand-authored top-level narrative files (`README.md`, `CONTRIBUTING.md`)", with two exclusions the widening must carry so it doesn't commit a category error:
   - **generated / gitignored files stay out** — `ROADMAP.md` (regenerated by `aiwf render roadmap`), `STATUS.md`, `WHITEBOARD.md`, `TODO.md`. Linting a generated artifact is wrong: if it drifts, the fix is to regenerate the source, not to file the skill's "report for human attention".
   - **append-only history stays out of the drift checks** — `CHANGELOG.md` *correctly* names removed features (that is the record it keeps), so drift checks #1 and #2 would false-positive on nearly every old entry.

   Orphan check #3 is structurally inapplicable to root files (`README.md` is the root everything links to), so only checks #1, #2, and #4 apply to the widened set.

## Why it matters

The skill's stated job is mechanical drift-catching, and the biggest drift class in a real docs tree — dead links between docs and into source — is exactly what it does not check, while it does check the harder, more heuristic symbol-existence case. A reader following a broken cross-link gets a 404; a contributor copying a doc's `aiwf …` example for a removed flag gets a failure the lint could have caught at the seam.

Because the skill ships to consumer repos, the `docs/`-only default means every consumer's `README.md` — its highest-traffic, most code-dense doc — goes unlinted unless the operator remembers to widen scope by hand. The exclusions matter as much as the inclusions: a naive "add the root files" would lint generated artifacts and flag correct changelog history as drift, turning a trustworthy report into noise. Keeping the widening principled — hand-authored source in, derived and append-only out — is what lets it stay default-on without eroding trust in the report.
